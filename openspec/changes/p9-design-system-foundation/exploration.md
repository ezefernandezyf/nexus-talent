## Exploration: P9 — Design System Foundation

### Current State

The project already has a significant design system ("The Signal") with:

- Anti-convergence rationale (rejects generic AI patterns)
- Dark-first strategy via `data-theme="light"` on `<html>`
- **Color palette**: OKLCH-native, Indigo (#5B4BFF / oklch 0.55 0.22 270) + Chartreuse (#C8F542 / oklch 0.92 0.19 130), with 8 surface tokens (Deep Space), 3 semantic tokens

### Approaches

1. **Clean slate** — Delete old DESIGN.md, run `/impeccable init` from scratch, rebuild everything
   - Pros: No legacy baggage, impeccable generates complete PRODUCT.md + DESIGN.md with full project identity
   - Cons: Loses 259 lines of DESIGN.md and 472 lines of index.css that already work well; rewrite of working font declarations, color tokens, component styles
   - Effort: **High**

2. **Evolve existing** — Keep current DESIGN.md and index.css, enhance with impeccable `audit`/`polish` + taste-skill guidance
   - Pros: Preserves all existing code, lowest risk, fastest time to ship
   - Cons: Misses the structured foundation that `/impeccable init` provides (PRODUCT.md, organized DESIGN.md sections); might carry forward suboptimal patterns
   - Effort: **Low-Medium**

3. **Hybrid (Recommended)** — Run `/impeccable init` to generate fresh PRODUCT.md + DESIGN.md, then merge proven token values (color palette, font choices, spacing/radii) from the current system back in. Use impeccable `extract` to pull reusable tokens. Apply redesign-existing-projects audit for components. Fix anti-patterns.
   - Pros: Gets impeccable's structured foundation (PRODUCT.md, identity-driven DESIGN.md), preserves proven decisions, fixes anti-patterns, aligns with the V1.2 redesign philosophy
   - Cons: Some merge effort required, manual reconciliation of old and new DESIGN.md content
   - Effort: **Medium**

### Risks

- **Merge conflicts with existing `index.css`**: The current file is complex (472 lines, 3 layers). Any rewrite of `@layer base` and `@theme` must preserve the dark/light mode token swaps or the app breaks.
- **Component breakage**: 26+ component files depend on current CSS classes (`.primary-button`, `.surface-panel`, `.field-surface`, etc.). Changing these class names or structure will cascade to feature components (AuthShell, AnalysisResultView, AppLayout, etc.).
- **Design identity dilution**: `/impeccable init` might suggest generic defaults. Must actively preserve the Indigo+Chartreuse identity that distinguishes Nexus Talent from generic SaaS.
- **Scope creep**: "No new components" (P9 scope) means we can't redesign Button, Input, Card, etc. But the CSS they depend on (`@layer components`) IS in `index.css`. We need to refactor the CSS structure without changing component behavior.
- **Timeline**: Proper merger of impeccable output + current system + taste-skill guidance + anti-pattern cleanup is a significant amount of work.

### Ready for Proposal

**Yes.** The exploration is complete. Key findings to convey to the orchestrator:

1. **The existing design system is surprisingly solid** — OKLCH tokens, dark-first, good font choices, working theme system. This is NOT a blank slate: we're evolving something that's already 70% there.
2. **The hybrid approach (Approach 3) is the right call**: run `/impeccable init` for the structured foundation, then merge back the proven token values. Don't throw away what works.
3. **The main gaps are**: missing typography scale, incomplete CSS reset, no z-index scale, basic skeleton components, and 1 gradient-text anti-pattern to fix.
4. **Biggest risk**: the `@layer components` refactoring touches everything. We can't ship broken buttons/inputs because we restructured the CSS without updating component imports.
5. **Recommendation for proposal phase**: Explicitly scope what CSS changes are allowed in P9 vs what waits for P10 component redesign. My suggestion: CSS layer refactoring in P9, component behavior stays identical until P10.
