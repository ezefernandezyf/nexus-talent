# Specification: Module 11 - Observability & Errors

## 1. Domain: Observability & Error Mapping

### Requirement: Centralized Logging Utility
The system MUST provide a centralized logging utility (`src/lib/logger.ts`) to replace raw `console` calls for consistency and future telemetry integration.

#### Scenario: Production Error Logging
- GIVEN the application is running in production mode
- WHEN an API or application error occurs
- THEN the logger MUST output the error details
- AND it MUST suppress `debug` and `info` logs by default unless explicitly enabled.

#### Scenario: Development Logging
- GIVEN the application is running in development mode
- WHEN the logger is invoked
- THEN the logger MUST output all log levels (`debug`, `info`, `warn`, `error`) to the console with appropriate formatting.

### Requirement: Domain Error Mapper
The system MUST include an error mapper (`src/lib/errors/error-mapper.ts`) that transforms technical errors into user-friendly messages.

#### Scenario: Mapping an AI API Rate Limit Error
- GIVEN the AI service returns a 429 Rate Limit error
- WHEN the error is passed to the error mapper
- THEN it MUST return a user-friendly message explaining that the service is busy and suggesting when to retry.

#### Scenario: Mapping a Zod Validation Error
- GIVEN a Zod validation fails during data parsing
- WHEN the ZodError is passed to the error mapper
- THEN it MUST return a clear message indicating which fields were invalid, avoiding raw JSON or stack traces in the UI.

## 2. Domain: UI & Feedback

### Requirement: Global React Error Boundary
The system MUST implement a global React Error Boundary (`src/components/ErrorBoundary.tsx`) to catch unhandled rendering exceptions and prevent white screens.

#### Scenario: Component Render Crash
- GIVEN a React component throws an unhandled error during rendering
- WHEN the Error Boundary catches the error
- THEN the system MUST display a fallback UI following the "Deep Space" visual guidelines (no solid borders, proper elevation, `surface-container` background).
- AND the system MUST log the error using the centralized logging utility.
- AND the fallback UI MUST provide a way for the user to recover or reload the application.

### Requirement: Global Toast Notification System
The system MUST integrate a global toast notification system to surface asynchronous process results (success/error).

#### Scenario: Async Mutation Failure
- GIVEN an asynchronous mutation (e.g., saving data via TanStack Query) fails
- WHEN the global query cache handler processes the error
- THEN the system MUST display an error toast using the user-friendly message generated by the error mapper.

#### Scenario: Async Mutation Success
- GIVEN an asynchronous mutation completes successfully where visual confirmation is required
- WHEN the operation resolves
- THEN the system MUST display a success toast following the established design system (e.g., "Glow Pulse" indicator, no heavy badges).
